## ADDED Requirements

### Requirement: Recurrence tab visibility in ReviewModal
The ReviewModal SHALL display a "Recurrence" tab pill for all events when the user has edit permissions. The tab SHALL appear between "Details" and "Attachments" in the tab order.

#### Scenario: Tab visible for event with no recurrence
- **WHEN** user opens ReviewModal for a single (non-recurring) event with edit permissions
- **THEN** the "Recurrence" tab pill is visible and clickable

#### Scenario: Tab visible for recurring event
- **WHEN** user opens ReviewModal for an event with an active recurrence pattern
- **THEN** the "Recurrence" tab pill is visible with an active-indicator dot

#### Scenario: Tab hidden for read-only users
- **WHEN** user opens ReviewModal without edit permissions
- **AND** the event has no recurrence pattern
- **THEN** the "Recurrence" tab pill SHALL NOT be displayed

#### Scenario: Tab visible read-only for recurring events without edit permissions
- **WHEN** user opens ReviewModal without edit permissions
- **AND** the event has an active recurrence pattern
- **THEN** the "Recurrence" tab pill SHALL be visible
- **AND** the tab content SHALL be read-only (no Create/Edit/Remove actions)

### Requirement: Empty state with Create Recurrence CTA
When the Recurrence tab is active and no recurrence pattern exists, the tab content SHALL display a centered empty state with a "Create Recurrence" button.

#### Scenario: Empty state display
- **WHEN** user clicks the Recurrence tab
- **AND** the event has no recurrence pattern
- **THEN** the tab shows a centered icon, explanatory text, and a "Create Recurrence" button

#### Scenario: Create Recurrence opens pattern modal
- **WHEN** user clicks "Create Recurrence" in the empty state
- **THEN** the existing RecurrencePatternModal opens with the event start date pre-filled

#### Scenario: After pattern creation, tab switches to management view
- **WHEN** user saves a new recurrence pattern in the modal
- **THEN** the empty state is replaced by the two-column management view
- **AND** the tab pill gains an active-indicator dot

### Requirement: Two-column management view layout
When a recurrence pattern exists, the Recurrence tab SHALL display a two-column layout with a sticky left column and scrollable right column.

#### Scenario: Left column content
- **WHEN** the management view is displayed
- **THEN** the left column (~350px, sticky) shows: pattern summary text, "Edit Pattern" button, interactive mini-calendar with color-coded dates, and a legend

#### Scenario: Right column content
- **WHEN** the management view is displayed
- **THEN** the right column (flex) shows: occurrence count header, optional filter bar, and a scrollable list of occurrence rows

#### Scenario: Responsive stacking
- **WHEN** the ReviewModal viewport width is insufficient for two columns
- **THEN** the layout SHALL stack vertically (left column content above right column content)

### Requirement: Occurrence list displays all series dates
The occurrence list SHALL show every date in the recurring series including pattern dates, ad-hoc additions, and exclusions.

#### Scenario: Pattern date row
- **WHEN** a date is generated by the recurrence pattern
- **THEN** the row shows: date, time range, assigned room(s), and conflict status icon (checkmark or warning)

#### Scenario: Ad-hoc addition row
- **WHEN** a date is an ad-hoc addition (not from the pattern)
- **THEN** the row is visually distinguished with a green accent and addition indicator
- **AND** a "Remove" action is available (if user has edit permissions)

#### Scenario: Excluded date row
- **WHEN** a date has been excluded from the pattern
- **THEN** the row shows with strikethrough styling and a red/muted accent
- **AND** a "Restore" action is available (if user has edit permissions)

#### Scenario: Conflict inline display
- **WHEN** an occurrence has a scheduling conflict
- **THEN** the row shows a warning icon with a brief conflict description
- **AND** clicking/expanding the row reveals full conflict details (conflicting event name, time, room)

### Requirement: Interactive mini-calendar in management view
The left column SHALL include an interactive mini-calendar that allows users to add ad-hoc dates and exclude pattern dates by clicking.

#### Scenario: Click pattern date to exclude
- **WHEN** user clicks a date that is part of the recurrence pattern
- **THEN** the date is added to exclusions
- **AND** the occurrence list updates to show the date as excluded

#### Scenario: Click non-pattern date to add
- **WHEN** user clicks a date that is not part of the pattern and not already added
- **THEN** the date is added as an ad-hoc addition
- **AND** the occurrence list updates to show the new addition row

#### Scenario: Click excluded date to restore
- **WHEN** user clicks a date that is currently excluded
- **THEN** the exclusion is removed
- **AND** the occurrence list updates to show the date as a normal pattern date

#### Scenario: Click ad-hoc date to remove
- **WHEN** user clicks a date that is an ad-hoc addition
- **THEN** the addition is removed
- **AND** the occurrence list removes that row

### Requirement: Edit Pattern reuses existing modal
The "Edit Pattern" button in the management view SHALL open the existing RecurrencePatternModal pre-populated with the current pattern.

#### Scenario: Edit Pattern opens modal
- **WHEN** user clicks "Edit Pattern"
- **THEN** RecurrencePatternModal opens with current frequency, interval, days, end date, additions, and exclusions

#### Scenario: Pattern changes reflect in management view
- **WHEN** user saves changes in the RecurrencePatternModal
- **THEN** the management view updates: pattern summary, mini-calendar, and occurrence list all reflect the new pattern

### Requirement: Remove Recurrence action
The management view SHALL include a "Remove Recurrence" action that clears the pattern and returns to the empty state.

#### Scenario: Remove recurrence with two-click confirmation
- **WHEN** user clicks "Remove Recurrence"
- **THEN** the button enters confirm state (text changes to "Confirm?", styled per app confirmation pattern)
- **AND** if confirmed within 3 seconds, the recurrence pattern is cleared
- **AND** the tab content returns to the empty state
- **AND** the tab pill active-indicator dot is removed

### Requirement: Details tab recurrence summary card
The recurrence section on the Details tab SHALL display a compact summary card with a link to switch to the Recurrence tab.

#### Scenario: Summary card for active recurrence
- **WHEN** the Details tab is active and the event has a recurrence pattern
- **THEN** the recurrence section shows: pattern summary text, occurrence count, additions/exclusions count, conflict count, and a "Manage Recurrence" link

#### Scenario: Manage Recurrence switches tab
- **WHEN** user clicks "Manage Recurrence" on the Details tab summary card
- **THEN** the active tab switches to "Recurrence"

#### Scenario: No recurrence on Details tab
- **WHEN** the Details tab is active and the event has no recurrence pattern
- **THEN** the recurrence section shows a compact invitation with "Set up on the Recurrence tab" link

### Requirement: Conflicts tab absorbed for recurring events
The standalone Conflicts tab SHALL be removed for recurring events. Conflict information SHALL be displayed inline in the Recurrence tab occurrence list.

#### Scenario: No separate Conflicts tab for series masters
- **WHEN** user opens ReviewModal for a series master event
- **THEN** there is no "Conflicts" tab in the tab bar
- **AND** conflict information is available in the Recurrence tab occurrence list

### Requirement: Occurrence list filtering
The occurrence list SHALL support filtering to help users find specific occurrences.

#### Scenario: Filter by type
- **WHEN** user selects a filter option (All, Added only, Excluded only, Conflicts only)
- **THEN** the occurrence list shows only rows matching the selected filter

#### Scenario: Default filter
- **WHEN** the Recurrence tab is first opened
- **THEN** the filter is set to "All" showing every occurrence
