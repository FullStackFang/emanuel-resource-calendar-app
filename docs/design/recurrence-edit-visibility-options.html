Design review · Recurrence edits

How should we surface recurrence changes on the event details page?

Today, when a requester edits an existing event, every changed scalar field gets a green diff card. The Recurrence tab is silent — pattern changes, new exclusions, and per-occurrence overrides are invisible to the approver. Three treatments below, faithful to the existing review-modal vocabulary.

Selected: — none —

Reference scenario

Same change set, rendered three ways.

  • Published event: Yoga · weekly Tue 10:00 AM · 14 occurrences
  • Pattern change: now Tue + Wed
  • New exclusion: Tue May 6
  • New override: Tue May 13 · 10:00 → 10:30
Option A · Recommended Effort · Medium

Tab marker + section diffs

A change indicator on the tab itself, plus inline diffs at every level of the tab — pattern, calendar, and per-occurrence rows.

Recommendation

Option A — but A and B are not mutually exclusive.

Option A is the only treatment that catches the most common case in production: a requester customizing one occurrence's time without changing the pattern. The pattern-level diff (Option B) compresses that into "+1 customization" — which costs the approver a manual diff against memory.

If engineering capacity is the constraint, ship B first as a one-day improvement (banner + tab dot using existing computeDetectedChanges), then layer on the per-row chips and dashed day ring from A. The two share the same green vocabulary, so they compose without redesign.
